# 存储原理

本章详细说明 PixivDownloader 把数据存在哪里、数据库如何记录文件位置，以及搬迁 / 备份时应该怎么做。

PixivDownloader 把数据分为两大类，**分开管理**：

- **作品文件**：你下载的插画、漫画、动图、小说等，全部存放在下载根目录（`download.root-folder`，默认 `pixiv-download/`）。
- **运行期数据**：配置、数据库、缓存、日志等，存放在工作目录下的 `config/`、`state/`、`data/`、`log/` 四个目录。

> [!NOTE]
> 下载根目录里**只有**作品文件。数据库、缩略图缓存等辅助数据一律不会写进去，因此你可以放心地把下载根目录放到大容量磁盘上，或单独同步 / 备份它。

---

## 工作目录总览

以程序所在目录（工作目录）为根，完整布局如下：

| 路径 | 内容 | 删除后果 |
|---|---|---|
| `config/config.yaml` | 主配置文件（[配置参考](/zh-cn/configuration)） | 重启后按默认值重新生成，自定义配置丢失 |
| `config/image_classifier.properties` | 图片分类工具的目录配置 | 分类目录设置丢失 |
| `state/setup_config.json` | 首次安装向导的结果：运行模式（solo / multi）、登录态等 | 需要重新走安装向导 |
| `state/batch_state.json` | 批量下载队列的断点状态 | 未完成的批量队列丢失 |
| `state/gui/` | GUI 引导、代理向导等一次性标记 | 部分引导会重新出现 |
| `state/download_root_marker.txt` | 下载根目录上次解析到的绝对路径（详见下文「符号根」） | 失去一次「配置改了但文件没搬」的启动检查 |
| `data/pixiv_download.db` | **SQLite 主数据库**（含 `-wal` / `-shm` 伴随文件） | 下载历史、画廊、收藏夹、计划任务等全部记录丢失（作品文件不受影响） |
| `data/collection_icons/` | 收藏夹自定义图标 | 图标恢复默认 |
| `data/gallery_thumbs/` | 画廊缩略图缓存 | 自动重新生成 |
| `data/tts/` | 听书（Edge TTS）的版本号缓存 | 自动重新获取 |
| `data/narration-voice/` | 多角色朗读的角色参考音频 | 对应角色失去音色克隆参考音 |
| `data/backfill/` | 回填工具记录的不可达作品清单 | 回填时会重新探测 |
| `log/` | GUI 与后端运行日志 | 仅影响排查问题 |
| `pixiv-download/`（即 `{下载根目录}`） | **全部作品文件** | 作品本体丢失 |

## 下载根目录的内部结构

| 路径 | 内容 |
|---|---|
| `{根}/{作品ID}/` | 单作品下载、URL 批量、搜索下载等的作品目录，每个作品一个文件夹 |
| `{根}/{画师名}/{作品ID}/` | **画师批量下载**的作品；R-18 / R-18G 作品会再嵌套一层 `R18/` 或 `R18G/` 子目录。`download.user-flat-folder: true` 时不建画师层，与上一行相同 |
| `{根}/{作品ID}/{文件名}_p0.webp` + `..._p0_thumb.jpg` | 动图（ugoira）合成后的 WebP 与首帧缩略图 |
| `{根}/novel-{小说ID}/` | 下载的单本小说（TXT / HTML / EPUB）及封面 |
| `{根}/artwork-series-{系列ID}/cover.{ext}` | 漫画系列封面 |
| `{根}/novel-series-{系列ID}/` | 小说系列封面与合订本文件 |
| `{根}/_archives/{token}.zip` | 多人模式配额打包、画廊批量导出生成的压缩包（有有效期，过期自动清理） |

作品目录内的图片文件名默认是 `{作品ID}_p{页码}.{扩展名}`，可在批量下载设置里用「文件名模板」自定义（如加入作者名、时间戳等）。

另外，**收藏夹可以指定独立的下载目录**：为收藏夹设置了专属下载目录后，勾选该收藏夹下载的作品会落到那个目录（可以在下载根目录之外），目录结构与上表相同。

## 数据库记录了什么

`data/pixiv_download.db` 是一个 SQLite 数据库（WAL 模式），保存除作品文件本体之外的全部记录：

- **下载历史**：每个作品 / 小说的标题、所在目录、页数、扩展名、下载时间、R-18 标记、作者、文件名模板等——画廊、判重（「已下载，跳过」）都靠它；
- **画廊数据**：标签、作者、系列、收藏夹及其成员、统计数字；
- **小说正文**：小说的原始内容直接存在数据库里（用于全文搜索、AI 翻译、重新导出），磁盘上的 TXT / EPUB 只是导出产物；AI 译文、听书的逐句朗读脚本也存在库中；
- **计划任务**：任务定义、运行水位、隔离重试队列（绑定的 Pixiv Cookie 同样存在库中，请妥善保管数据库文件）;
- **疑似重复检测**：每页图片的感知哈希。

> [!WARNING]
> 数据库与作品文件是**两份独立的数据**：删除数据库不会删除作品文件，但所有历史记录、画廊与判重信息都会丢失；反过来只删文件不删记录，画廊里就会出现打不开的条目。备份时两者（连同 `-wal` / `-shm` 伴随文件）应一起备份。

## 路径是怎么记录的：前缀编码与符号根

数据库需要记住每个作品所在的目录。如果直接存几万条绝对路径，既浪费空间，目录一旦搬走也会全部失效。PixivDownloader 用**前缀编码**解决这个问题：

### `{N}` 前缀引用

所有目录列（作品目录、移动目录、小说目录、系列封面目录、收藏夹下载目录）存成：

```
{N}/相对路径        例如 {3}/novel-12345
```

`N` 指向 `path_prefixes` 表中登记的一条绝对路径前缀（如 `D:\Pictures\pixiv-classified`）。读取时把前缀拼回去得到完整路径。这样每个根目录的绝对路径只存一份；用 GUI 的「迁移下载目录」改写这一份，所有引用它的记录就同时指向新位置。

### `{0}` 符号根：跟随软件目录

`{N}` 仍然钉死在某个绝对路径上——把整个软件文件夹拷到新电脑 / 新盘符后，记录里的 `D:\旧位置\...` 就全部失效了。为此引入了**符号根 `{0}`**：

- 当 `download.root-folder` 配置为**相对路径**（默认的 `pixiv-download` 就是），下载根目录下的记录会编码成 `{0}/相对路径`；
- `{0}` 不登记任何绝对路径，每次启动时**动态解析**为「当前软件目录 + 配置的相对路径」；
- 因此**把整个软件文件夹（连同里面的下载根目录）复制 / 搬迁到任何新位置，画廊和下载历史都能直接定位文件**，无需任何修复操作——这正是默认安装的推荐用法；
- 旧版本留下的绝对路径记录会在升级后首次启动时自动迁移为 `{0}` 格式，无需手动处理。

如果你把 `download.root-folder` 配置成**绝对路径**（例如指向另一块磁盘），符号根不启用，记录回到 `{N}` 方式——此时搬迁下载目录就需要走 GUI 的「迁移下载目录」工具。

### 防呆检查：改了配置但没搬文件怎么办

`state/download_root_marker.txt` 记录了上一次运行时下载根目录解析到的绝对路径，配合两道防护：

- **GUI 配置页拦截**：在「配置 → 下载 → 下载根目录」修改路径、且数据库中存在「跟随软件目录」的旧记录时，保存前会弹窗说明后果——继续保存会把旧作品的记录固定为当前绝对路径（失去随软件整体搬迁的能力），点「确定」自动完成固定后再保存，点「取消」放弃本次修改。
- **启动检查**：如果绕过 GUI 直接手改了 `config.yaml`（或换了启动目录），启动时会发现解析位置与上次不一致：
  - 下载根目录已改为**绝对路径**但数据库仍有「跟随软件目录」的旧记录 → GUI 管家启动后会弹出修复窗口，引导你确认这些作品实际所在的旧下载根目录（已预填上次记录的位置），确认后自动把旧记录固定到该路径；
  - 仍是相对路径但旧位置还有文件、新位置为空 → 多半是文件没搬，启动日志持续警告并提示用「迁移下载目录」修复，直到处理完毕；
  - 新位置文件齐全 → 视为整体搬迁的正常跟随，只记一条提示。

### GUI「迁移下载目录」如何处理符号根

迁移工具（GUI 管家 → 状态页 → 迁移下载目录）会把「当前下载根目录（跟随软件目录）」置顶显示完整解析路径。为它填入新目录后，会根据新目录位置询问是否同步修改 `config.yaml`：

| 新目录位置 | 同步改配置？ | 结果 |
|---|---|---|
| 仍在软件目录内 | 是（推荐） | 只把 `config.yaml` 改成新的**相对路径**，数据库记录不动、继续跟随软件目录 |
| 仍在软件目录内 | 否 | 数据库记录固定为新目录的绝对路径，配置不变（新下载仍进旧目录） |
| 在软件目录外 | 是 | 配置改为新目录的绝对路径，数据库记录一并固定过去 |
| 在软件目录外 | 否 | 仅数据库记录固定到新目录，配置不变（新下载仍进旧目录） |

> [!NOTE]
> 迁移工具只改数据库记录，**不会移动磁盘上的文件**——请先自行把文件搬到新目录（新目录必须已存在），再执行迁移。修改下载根目录后需重启服务生效。

## 搬迁与备份指南

**整体搬迁（推荐方式）**
保持 `download.root-folder` 为相对路径（默认即是），直接把整个软件文件夹拷贝 / 移动到新位置（新电脑、新磁盘均可），启动即用，画廊与历史记录自动跟随，无需任何额外操作。

**只搬下载目录**
1. 关闭服务，把下载根目录的文件移动到新位置；
2. 启动 GUI 管家 → 状态页 → 迁移下载目录，按上表选择对应分支；
3. 按提示重启服务。

**备份**
最少需要：`config/` + `data/`（含 `-wal` / `-shm`）+ 下载根目录。`state/` 可选（丢失只需重新走一次安装向导 / 重新登录）。

> [!WARNING]
> 不要在没有搬文件的情况下手工修改 `config.yaml` 的 `download.root-folder`。相对路径模式下这会让所有历史记录跟着解析到新的空目录（启动日志会警告）；正确做法始终是「先搬文件，再用迁移工具改记录 / 配置」。
